1. AI시황 시스템 소개
1.1 시스템 개요
AI 시황 시스템은 금융 시장 전반의 흐름과 주요 변동 요인을 인공지능 기반 분석을 통해 사용자에게 제공하는 서비스입니다. 국내 시장을 중심으로 최근 발생한 주요 이슈, 테마 및 업종 동향, 종목별 시황 등을 종합적으로 분석하여 제공함으로써 사용자가 현재 시장 상황을 빠르게 이해할 수 있도록 지원합니다.
본 서비스는 실시간 뉴스 데이터와 KRX 시장 데이터를 기반으로, Azure OpenAI(GPT-4.1) 모델을 활용하여 시황 콘텐츠를 자동 생성합니다. 생성된 콘텐츠는 장중 약 10~30분 주기로 갱신되고, KOSPI 지수가 ±4% 이상 변동하는 경우 긴급 시황 콘텐츠가 생성되어 시장 상황을 신속하게 전달하도록 설계되어 있습니다.

1.3 서비스 접속 정보 및 환경
AI 시황 서비스는 웹 기반 서비스로 제공되며 다음 URL을 통해 접속할 수 있습니다.

- 운영 환경: https://marketai.nhsec.com/ms/aimc
운영 담당자는 테스트 앱 외에도 해당 URL을 통해 서비스 화면 및 시황 생성 결과를 확인할 수 있습니다.

Azure Databricks는 내부 프록시 설정 후에 다음 URL로 접속합니다
- 접속 경로: 내부망 PC → 내부 프록시 설정 → Databricks URL 접속.
- 개발 환경: https://adb-1262864985660652.12.azuredatabricks.net
- 운영 환경: https://adb-503793798960427.7.azuredatabricks.net

데이터 수집서버는 Azure VM에서 가동되며 Bastion Host을 거쳐 접속합니다
- 접속 경로: 로컬 PC → Bastion Host (Gateway) → 수집 서버 (Target)
- 개발 환경 접속 방법:
로컬 PC에서 PowerShell 실행
Bastion Host 접속: az ssh vm -n vm-hub-bst-shd -g rg-hub-shd --port 62234
수집 서버(Azure VM) 내부 접속: az ssh vm -n nh-vm-market-dev -g rg-dataings-dev –port 62234
- 운영 환경 접속 방법:

Bastion Host 접속: az ssh vm -n vm-hub-bst-prd -g rg-hub-shd --port 62234
수집 서버(Azure VM) 내부 접속: az ssh vm -n nh-vm-market-prd-01 -g rg-dataings-prd –port 62234 (서버1)
az ssh vm -n nh-vm-market-prd-02 -g rg-dataings-prd –port 62234 (서버2)

Azure Portal 접속 및 Apps 실시간 로그 조회는 외부망에서 하단의 URL로 접속 후 확인할 수 있습니다.

1.4 메뉴 구성 개요
생성된 시황 데이터는 Backend API를 통해 조회 및 Redis 갱신이 되며, Frontend Web에서 사용자에게 표시됩니다. 또한 ADLS를 통해 외부 시스템 연동용 CSV를 내보내는 작업도 진행합니다.
- Backend API: 시황 데이터 조회/갱신 알림 (Azure Container Apps)
- Frontend Web: 웹 기반 시황 화면 (Azure Container Apps)
- ADLS: 외부 시스템 연동용 CSV 내보내기 (네이버클라우드 커뮤니티 전송)

2.2 데이터 수집 파이프라인
AI 시황 시스템의 데이터 수집은 실시간 시장 데이터, 외부 데이터로 구성됩니다.

2.2.1 실시간 시장 데이터 수집
실시간 시장 데이터는 UDP 기반 수집 서버를 통해 수신됩니다.
UDP 수집 서버 (VM, 포트 30101~30107)
 → ADLS (landing/port=301xx/*.jsonl)
 → Databricks 00_ingest_bronze
 → Bronze 테이블 적재
수집된 데이터는 JSONL 형식으로 저장되며 Databricks에서 Bronze 레이어로 적재됩니다.

2.2.2 SNS 및 외부 데이터 수집
SNS 및 외부 데이터는 Container App 기반 수집 모듈을 통해 수집됩니다.
sns-int (Container App, 스케줄러)
 → sns-ext (Container App, 수집 API)
 → ADLS (landing/SNS/perplexity|telegram|quartr|ezar/)
 → Databricks 03/04/06_*_ingest_batch
 → Bronze 테이블 적재
현재 수집 대상 데이터는 Perplexity 뉴스, Telegram, Quartr, Ezar로, 해당 데이터들은 이후 종목 뉴스 분석 및 시황 생성 과정에서 활용됩니다.
2.3 데이터 발송 파이프라인
2.3.1 주요 국내 시황 요약 발송 
Databricks에서 생성된 국내 시황 요약 데이터는 외부 커뮤니티 서버로 전달됩니다.
Databricks 65_macro_summary_export.py

해당 과정은 Databricks 작업 완료 후 자동으로 호출됩니다.

3.1.2 수집 방식
UDP 데이터 수집은 다음과 같은 3단 파이프라인 구조로 동작합니다.
1) UDP 데이터 수신
2) 큐 기반 버퍼링 (포트당 약 7,500건)
3) 워커 스레드 기반 데이터 처리

3.1.3 처리 방식
위와 같이 수집된 데이터는 다음 방식으로 처리됩니다.

&lt;UDP 수신 데이터 구조&gt;
- 공통 처리 흐름: UDP 수신 → Queue → 디코딩 → 채널 분류 → 파서 호출 → JSONL/CSV 싱크 → ADLS 업로드
- 1분 단위 JSONL 파일 로테이션 및 ADLS 업로드
- 뉴스/공시(30106) 처리 파이프 라인
원시 UDP 패킷 (바이너리)
  ▼
① MUDP 헤더 8바이트 파싱
  │  data_cls_code[0]: 'N'=뉴스, 'E'/'F'=공시
  │  comp_type[1]:     'Y'=LZO 압축, 'N'=비압축
  │  content_mode[5]:  'O'=100기반 오프셋, 그 외=0기반 오프셋
  ▼
② LZO 압축 해제 (comp_type == 'Y'인 경우)
  │  라이브러리: lzokay
  │  body_bytes = lzokay.decompress(body_bytes)
  ▼
③ EUC-KR 디코딩 + 널바이트(\x00) 제거
  ▼
④-A 바이너리 구조체 파서 (1순위: binary_parser.py)
  │  find_n_news_start()로 구조체 시작점 동적 탐색
  │  N_NEWS_MM 구조체 순차 언패킹
  ├─ 실패 시 ─▶

  │  content_mode에 따라 고정 오프셋 슬라이싱
  ▼
⑤ 뉴스 본문 정규화 (fgubun 규칙 적용)
  │  HTML 태그 제거 + fgubun별 substring/regex 적용

3.3 데이터 저장소 구조
수집된 데이터는 ADLS와 데이터 레이어 구조를 통해 저장되고, 이후 데이터 처리 및 분석 과정에서 활용됩니다.
3.3.1 ADLS 구조
AI 시황 시스템은 Azure Delta Lake Storage를 데이터 저장소로 사용합니다.
주요 저장 영역은 landing/ 입니다.
- 데이터 유형별 주요 경로
landing/port=301xx/            실시간 시장 데이터
landing/SNS/                   SNS 수집 데이터
landing/snd/                   시황 요약 발송 데이터
* Landing 영역은 원천 데이터 저장 영역으로 사용되며 이후 Databricks 파이프라인에서 처리됩니다.

3.3.2 데이터 레이어 구조
Databricks에서는 데이터 레이크 아키텍처를 기반으로 다음과 같은 레이어 구조를 사용합니다.
데이터는 처리 단계에 따라 다음과 같은 Bronze / Silver / Gold 계층으로 관리됩니다.
- Bronze Layer: 원천 데이터 저장
- Silver Layer: 가공 및 정제 데이터
- Gold Layer: 시황 생성 데이터

4. 데이터 처리 및 AI 생성
데이터 처리 작업은 Azure Databricks 환경에서 수행됩니다.

4.3 AI 시황 생성 항목 및 기준

주요 생성 콘텐츠는 다음과 같고, Gold Layer에서 다음과 같은 시황 데이터가 생성됩니다.

2) 최근 이슈 (A200_MARKET_EVENTS)
시장에 영향을 준 뉴스 및 이벤트를 기반으로 생성되는 시황 콘텐츠입니다.

3) 주요 테마 및 업종 (A300_THEME_MARKET)
시장 내 주요 테마 및 업종 동향을 중심으로 생성되는 시황입니다.

4) 종목 시황 (A400_STOCK_NEWS)
개별 종목을 기준으로 생성된 시황입니다.

5) 긴급 시황 (A000_URGENT_MARKET)
시장 변동성이 급격하게 확대되는 경우 생성되는 특수 시황 콘텐츠입니다.

5.2.2 결과 수정 및 웹페이지 반영 방법
AI 시황 결과가 MTS에 표시되기까지의 데이터 흐름은 다음과 같습니다:
Gold 테이블 값 수정  →  refresh API 호출  →  Redis 캐시 갱신 →  MTS 화면 반영

AI 시황 결과는 Databricks Gold 테이블에 저장되어 있고, 백엔드가 이를 Redis에 캐싱하여 MTS에 제공합니다. 결과를 수정하려면 Gold 테이블의 값을 수정한 뒤, 반드시 refresh API를 호출해야 MTS 화면에 반영됩니다.

방법 1: Databricks 노트북에서 수정
1-1. Gold 테이블 값 수정
Databricks Workspace 접속
SQL Editor 열기 (또는 노트북에서 SQL 셀 사용)
수정 대상 데이터 확인:
해당 행의 값 수정
-- 예: 특정 행의 본문 내용 수정
UPDATE nh_ai_prd.gold.a100_macro_market
SET summary = '수정된 시황 내용...'
WHERE date = '2026-03-23' AND id = '대상_ID';

-- 예: 특정 행 삭제
DELETE FROM nh_ai_prd.gold.a300_theme_market
WHERE date = '2026-03-23' AND theme_name = '삭제할_테마명';

1-2. refresh API 호출 (필수)
Gold 테이블 수정 후, 반드시 백엔드 refresh API를 호출해야 MTS에 반영됩니다.
Databricks 노트북 셀에서 실행 (Gold 스크립트의 마지막 셀):
import requests

# 개발 환경
url = "https://app-trend-int-dev.purplerock-9761bdbb.koreacentral.azurecontainerapps.io/api/market/refresh"

# 운영 환경
# url = "https://app-trend-int-prd.ambituousbush-1e7439b2.koreacentral.azurecontainerapps.io /api/market/refresh"

# 수정한 카테고리에 맞게 지정
response = requests.post(url, json={"category": "macro"}, timeout=15)
print(response.json())

1-3. refresh 후 동작
백엔드(trend-int)가 refresh 요청을 수신하면:
해당 카테고리 오늘자 Redis 캐시 삭제
Databricks Gold 테이블에서 수정된 데이터를 재조회
Redis에 새 데이터 저장 (TTL: 자정까지)
urgent인 경우 SSE로 MTS에 즉시 푸시

Ex) POST /api/market/refresh  { "category": "macro" }
  → Redis 오늘자 macro 키 삭제
  → nh_ai_prd.gold.a100_macro_market 재조회
  → Redis에 새 데이터 저장
  → MTS가 다음 API 호출 시 수정된 데이터 수신

방법 2: 관리자 대시보드에서 수정
관리자 대시보드에서 Gold 테이블의 개별 행을 UI로 직접 수정할 수 있으며, 저장 시 자동으로 refresh가 호출됩니다. (권한 ops 또는 admin 역할 필요)
1. 사이드바 &gt; 데이터 관리 &gt; AI 시황 데이터 관리
2. 상단에서 수정할 카테고리 테이블 선택
3. 날짜/조건으로 필터링하여 수정할 행 찾기
4. 해당 행 클릭 후 우측의 작업 아이콘으로 내용 편집 (또는 삭제)
5. 수정 이후 저장 버튼 클릭

저장(수정/삭제) 시 관리자 대시보드가 자동으로 다음을 수행합니다. 
  → Databricks에 UPDATE/DELETE SQL 실행
  → 성공 시 POST MARKET_REFRESH_API_URL (백엔드 refresh 자동 호출)
  → Redis 캐시 갱신
→ 화면에 "캐시 갱신 완료" 또는 "캐시 갱신 실패" 토스트 표시

방법 1과 달리 수동으로 refresh를 호출할 필요가 없습니다. 저장 시 테이블 → 카테고리 자동 매핑되어 refresh가 호출됩니다. 환경별 refresh 대상 URL은 관리자 대시보드의 환경변수 MARKET_REFRESH_API_URL로 설정됩니다
환경
MARKET_REFRESH_API_URL
개발
https://app-trend-int-dev.purplerock-9761bdbb.koreacentral.azurecontainerapps.io
운영
https://app-trend-int-prd.ambitiousbush-1e7439b2.koreacentral.azurecontainerapps.io

* 주의사항
방법 1 사용 시 refresh 호출을 실행해야 MTS에 반영됩니다. Gold 테이블만 수정하고 refresh를 호출하지 않으면 MTS에 반영되지 않습니다. 이 경우, 다음 자정 자동 갱신까지 기다려야 합니다.
자정 자동 갱신: 매일 KST 자정에 백엔드가 전체 Redis 캐시를 클리어하고 새로 프리로드합니다. refresh를 놓쳤더라도 다음 날에는 자동 반영됩니다.
긴급 시황(urgent)은 SSE 브로드캐스트를 통해 MTS에 실시간 푸시됩니다. 다른 카테고리는 MTS가 다음 API 호출 시 갱신된 데이터를 수신합니다.
관리자 대시보드의 MARKET_REFRESH_API_URL이 미설정이면 수정은 Databricks에 반영되지만 캐시가 갱신되지 않아 MTS에 즉시 반영되지 않습니다. 배포 시 반드시 설정을 확인하세요.

5.4.2 시스템 구성요소별 장애
1) Databricks Job 오류
Databricks Job 실행 오류가 발생하면 시황 생성 작업이 정상적으로 수행되지 않을 수 있습니다.
Job 실행 상태 확인 후 재실행하거나, SQL Warehouse 실행 로그를 확인 후 담당 부서에 문의하여 쿼리 실행 강제 종료를 하는 등의 조치를 진행합니다.

- Job 실행 상태 확인 및 재실행
Databricks 워크스페이스 &gt; Workflows &gt; Jobs 메뉴에서 해당 Job의 실행 이력(Runs)을 확인합니다.
각 Run의 상태(Succeeded / Failed / Timed Out / Cancelled)를 확인하고, Failed인 경우 Run 상세 화면에서 에러 로그를 확인합니다.
일시적인 오류(네트워크 타임아웃, 리소스 부족 등)인 경우, Run Now 버튼으로 해당 Job을 수동 재실행합니다.

2) SQL Warehouse 쿼리 실행 오류
Job 내부에서 SQL 쿼리 실행 중 오류가 발생하거나 쿼리가 장시간 응답하지 않는 경우(MTS 화면에서 시황 무한 로딩), SQL Warehouse 상태를 확인해야 합니다.

확인 경로: Databricks 워크스페이스 &gt; SQL &gt; SQL Warehouses 메뉴 진입
Warehouse 상태 확인: Running / Starting / Stopped 상태를 확인합니다. Stopped 상태인 경우 자동 시작이 되지 않았을 수 있으므로 수동으로 Start합니다.
쿼리 이력 확인: 해당 Warehouse를 클릭 &gt; Query History 탭에서 최근 실행된 쿼리의 상태(Running / Finished / Failed / Cancelled)를 확인합니다. 
- 장시간 Running 상태인 쿼리가 있으면 해당 쿼리를 Cancel하여 강제 종료합니다.
- Failed 쿼리는 에러 메시지를 확인하여 원인을 파악합니다.
리소스 모니터링: Warehouse 상세 화면의 Monitoring 탭에서 동시 실행 쿼리 수, 대기열 상태, 리소스 사용량을 확인합니다. 쿼리 병목이 발생할 경우 담당 부서에 Warehouse 스케일업 또는 쿼리 강제 종료를 요청합니다.

3) MTS 서비스 화면 오류
웹 서비스(MTS)에서 시황 데이터가 표시되지 않거나 빈 화면이 나타나는 경우, 다음 항목을 점검합니다.

- Redis 캐시 점검
시황 생성이 정상 완료되었으나 웹 화면에 반영되지 않는 경우, Redis 캐시 갱신이 실패했을 수 있습니다.
관리자 대시보드에서 해당 카테고리의 데이터를 조회하여 Gold 테이블에 데이터가 정상 존재하는지 확인합니다.
데이터가 존재하면 관리자 대시보드의 데이터 매니저에서 해당 항목을 수정(내용 변경 없이 저장)하면, 백엔드에 직접 POST /api/market/refresh를 호출하여 캐시를 수동으로 갱신합니다.

- 백엔드 서버 점검
백엔드 서버(app-trend-int-prd)가 정상 동작하는지 확인합니다.
Azure Portal에서 해당 Container App의 상태 및 로그 스트림을 확인합니다.
Redis 연결 오류, Databricks 쿼리 타임아웃 등의 에러가 발생하는 경우 해당 서비스의 연결 정보와 상태를 점검합니다.

- 프론트엔드 서버 점검
프론트엔드(app-trend-ext-prd)는 Nginx 기반으로 구동됩니다.
웹 페이지 자체가 로딩되지 않는 경우 Azure Portal에서 해당 Container App의 상태를 확인하고, 필요시 재시작합니다.

6. 운영 이슈 및 로그 확인
6.1 LLM 로그 조회 쿼리 및 차단 로그 확인
시황 생성 과정에서 Azure OpenAI(AOAI)를 호출할 때마다 요청/응답 내용과 보안 검증 결과가 nh_ai_prd.logs.LLM_REQUEST_RESPONSE 테이블에 자동으로 기록됩니다. 운영 담당자는 이 로그를 통해 LLM 호출 상태, 오류, 개인정보(PII) 탐지, 콘텐츠 필터 차단 등을 확인할 수 있습니다.

LLM 호출 결과에 따라 다음 유형으로 분류되어 기록됩니다.
LOG_TYPE

false

LLM 호출 시 다음 3단계의 보안 필터링이 적용됩니다.
1) PII 필터 (입력 전처리)
LLM에 요청을 보내기 전, ‘pii_filter.py’를 통해 입력 텍스트에서 개인정보를 탐지하고 마스킹합니다.
탐지 대상: 주민등록번호, 외국인등록번호, 여권번호, 운전면허번호, 카드번호, 계좌번호, 전화번호, API 키, 비밀번호
심각도 수준: LOW / MEDIUM / HIGH / CRITICAL
기본 동작: PII 감지 시 마스킹 후 LLM에 전송 (CRITICAL이라도 기본적으로 차단하지 않고 마스킹만 수행)
마스킹 이벤트는 별도로 nh_ai_prd.logs.SECURITY_EVENTS 테이블에도 기록됩니다.

2) AOAI 콘텐츠 필터 (Azure 서비스 측)
Azure OpenAI 서비스의 내장 콘텐츠 안전 필터로, 요청/응답에 대해 hate, self_harm, sexual, violence 카테고리를 검사합니다.
필터 위반 시 HTTP 400 에러와 함께 응답이 차단되며, CONTENT_FILTER_BLOCKED로 기록됩니다.

3) PII 필터 (출력 후처리)
LLM 응답에 대해서도 PII 탐지 및 마스킹을 수행합니다.
출력 단계에서는 차단 없이 마스킹만 적용됩니다.
추가로 API 키, 엔드포인트 URL, 모델명 등 시스템 정보 노출 패턴도 제거됩니다.

데이터브릭스 노트북 또는 SQL Editor에서 쿼리를 실행하여 LLM 로그를 조회할 수 있습니다. 관리자 대시보드에서는 코드 작성 없이 UI를 통해 LLM 로그와 보안 현황을 확인할 수 있습니다.
관리자 대시보드의 경우, 서비스 운영 관리 탭에서는 아래 내용을 확인할 수 있습니다. 자세한 내용은 관리자 대시보드 매뉴얼을 참고해주시기 바랍니다.

1) LLM 호출 로그
LLM 로그 목록: 프롬프트 ID 및 기간 기준으로 요청/응답 미리보기
LLM 호출 통계: 기간별 프롬프트 ID별 호출 횟수 및 에러 횟수 집계
LLM 로그 단건 상세: 특정 호출의 전체 REQUEST/RESPONSE, PII/필터/차단 컬럼 전부 확인

2) 보안 모니터링 화면
보안 일별 통계: 전체 호출 대비 PII 감지, 콘텐츠 필터, 차단 건수 추이
PII 유형별 건수: 감지된 개인정보 유형(주민번호, 카드번호 등)별 현황
콘텐츠 필터 카테고리: 필터링된 콘텐츠 카테고리(hate, violence 등)별 현황
차단 내역 목록: 차단된 요청의 사유, 프롬프트 ID, 요청 미리보기

3) SQL 실행기 (SQL Executor)
관리자 대시보드의 SQL 실행기 화면에서 위 쿼리들이 템플릿으로 사전 등록되어 있어, 클릭만으로 바로 실행할 수 있습니다.
'LLM 호출 로그', '보안 로그' 카테고리에서 원하는 템플릿을 선택하거나, 직접 SQL을 작성하여 실행할 수 있습니다.

6.2 주요 운영 이슈 및 특이사항
1) KRI1(국내 지수) 종가 데이터 이상값
현상: KRI1 패킷에서 장 종료 후 전송되는 종가 데이터는 BSOP_HOUR 값이 888888로 수신됩니다. 이 종가 패킷이 들어오는 시점은 장마감 15:30 이후 전송되며, 따라서 UDP Collector를 거쳐 nh_ai_prd.bronze.KRI1_RAW에 적재되는 시점이 장 종료(15:30) 직후가 아닌, 수 분 이상 지연될 수 있습니다.

영향: 시황 생성 프로그램들이 KRI1 데이터를 조회할 때 BSOP_HOUR 기준 내림차순으로 최신 1건을 가져오는 로직을 사용합니다. 
60_macro_market.py (306행): Window.partitionBy("BSTP_CLS_CODE").orderBy(F.col("BSOP_HOUR").desc()) → KRI1_SILVER에서 최신 1건 조회
81_urgent_market_trigger.py (176행): orderBy(F.col("BSOP_HOUR").desc()).limit(1) → 전일 종가 조회
11_kri1_silver.py (93행): orderBy(F.col("BSOP_HOUR").desc()) → 일자별 마지막 시점 집계
장 종료 직후에 실행되는 시황 배치(예: 15:30~15:40 사이)에서는 종가(BSOP_HOUR = 888888) 레코드가 아직 적재되지 않아, 장중 마지막 시세가 조회될 수 있습니다. 종가 기반의 정확한 시황이 필요한 경우, 배치 실행 시간을 종가 적재 이후로 충분히 여유 있게 설정하거나, 종가 레코드 존재 여부를 확인한 후 실행하는 것을 고려해야 합니다.

2) 수능일 및 새해 개장식 (특수 장운영일) 이슈

프론트엔드(mts-app)는 Nginx 기반으로 구동되며, 백엔드 API로의 프록시 통신 시 TLS 1.2/1.3 프로토콜을 사용합니다.
Nginx 설정(nginx.conf.template)에서 proxy_ssl_verify on 옵션이 활성화되어 있어, 백엔드 SSL 인증서가 만료되면 API 프록시가 실패합니다.
신뢰 인증서 경로: /etc/ssl/certs/ca-certificates.crt (컨테이너 내장 CA 번들)

SSL 인증서 갱신 시 확인사항
*.nhqv.com 도메인 SSL 인증서의 만료일을 사전에 확인합니다. (26.03.25 기준 만료일 27.02.24)
인증서 갱신은 Azure Portal의 해당 리소스(Container App) &gt; 네트워킹 &gt; 사용자 지정 도메인 메뉴에서 진행합니다.

7.2.2 최근 이슈 (시장이벤트) — 40_market_events.py → A200_MARKET_EVENTS
최근 뉴스에서 주요 시장 이슈를 3건 추출하고, 각 이슈의 상세 내용을 LLM으로 요약 생성합니다.
뉴스 선정 조건:
시간 범위: 최근 30분 이내 적재된 뉴스 (_INGEST_TS 기준)
GPT 점수 필터: GPT01_AD_POST_SCORE &lt; 70 (광고 제외), GPT03_MARKET_POST_SCORE &gt;= 50, GPT04_CONTENT_QUALITY_SCORE &gt;= 50
중요도 점수: GPT02 + GPT03 + GPT04 합산 기준 내림차순 정렬
상위 200건 선정 → 제목 정규화 후 중복 제거 → LLM에 전달
생성 프로세스 (2단계 LLM):
이슈 추출 (news_market_event 프롬프트): 뉴스 헤드라인 목록에서 3대 시장 이슈(issue_title_01~03) 선정. 기존 당일 이슈와 동일 이슈면 reuse_id로 연결하여 뉴스 누적
상세 생성 (news_market_event_content 프롬프트): 각 이슈별로 관련 뉴스 본문(최대 5건, 각 500자)을 취합하여 상세 내용(EVENT_DETAIL) 생성
표시 데이터: EVENT_TITLE(이슈 제목), EVENT_DETAIL(상세 내용), NEWS_IDS/TITLES(관련 뉴스), MAIN_NEWS_ID(GPT03 최고점 핵심 뉴스), DISPLAY_CNT(동일 이슈 누적 표시 횟수)
갱신 주기: Databricks Job 스케줄에 따라 30분 간격 (영업일 판단 후 실행)

7.2.3 주요 테마/업종(테마 시황) — 50_theme_market.py → A300_THEME_MARKET 
인포스탁(IFS) 테마 코드 기반으로 테마별 등락률, 거래대금, 시가총액을 집계하고, 관련 뉴스를 기반으로 테마 시황 문구를 LLM으로 생성합니다.
테마 선정 기준:
데이터 소스: EPDIW_IFS_TMA_CD(인포스탁 테마코드) + EPDIW_IFS_TMA_IEM(테마-종목 매핑)
각 테마의 구성종목에 대해 KRX1_RAW(국내주식 시세) 조인 → 종목별 등락률(전일대비), 거래대금, 시가총액 집계
종합점수(TOTAL_SCORE) 계산식: 
각 테마의 등락률 절대값, 거래대금, 시가총액에 대해 Z-Score 표준화
TOTAL_SCORE = RETURN_SCORE × 0.8 + (VOLUME_SCORE − MARKETCAP_SCORE) × 0.2
등락률 비중 80%, 거래대금 대비 시가총액 비중 20%
TOTAL_SCORE 상위 20개 테마 선정
버블 사이즈(BUBBLE_SCALE) 계산:
상위 20개 테마의 등락률 절대값을 기준으로 분위수(Quantile) 5등분
q20, q40, q60, q80 경계값을 산출하여 해당 테마의 등락률 절대값이 어느 구간에 속하는지 판단
뉴스 선정 및 시황 생성 (2단계 LLM):
뉴스 후보: 최근 30분 이내, GPT01 &lt; 70(광고 제외), GPT04 &gt; 0(본문품질) 필터
종목 등락률 기반 필터링: 테마 등락률과 반대 방향이고 절대값 2% 이상인 종목의 뉴스 제외
대표뉴스 선별 (theme_rep_news_select): 테마 설명·등락률·종목 등락률을 고려하여 최대 10건 선별. 홍보성/비영향 기사 배제
시황 생성 (theme_issue_summarize): 선별된 뉴스 + 직전 시황 + 종목별 테마 선정 사유를 기반으로 2~3문장 Fact-Based 서술
연관 종목 표시:
대표뉴스의 연관종목 + 테마 내 고변동 종목(등락률 절대값 3% 이상, 상위 5개) + 직전 시황 연관종목(최대 3개) 누적
뉴스 ID/제목은 당일 내 직전 시황에서 누적 유지 (최신순 정렬)
갱신 주기: 30분 간격 (배치 시간: 08:30~16:30). 실행마다 기존 당일 테마의 실시간 필드(FLUCTUATION_RATE, TRANSACTION_AMT, MARKET_CAP, BUBBLE_SCALE, DIRECTION, NEWS_RETURNS)도 갱신
7.2.4 종목 시황 — 70_stock_news.py → A400_STOCK_NEWS

 버튼 클릭 시 주요 테마/업종에 대한 가이드를 확인할 수 있습니다. 또한 [버블차트] 클릭 시 해당 테마의 시황을 확인하는 페이지로 이동하며, 해당 페이지에서는 테마에 대한 변동률, 거래대금, 구성종목, 시가총액 등 상세 내용을 확인할 수 있습니다.
이동한 테마 시황 상세 페이지에서 날짜를 변경하면 변경한 날짜에 해당하는 테마 시황을 확인할 수 있습니다. 단, 테마 시황이 존재하는 날짜만 표기됩니다. 추가적으로, 테마 이슈 중 기사를 클릭하면, 해당 기사 전문을 확인할 수 있고, 종목 칩을 클릭하면 종목 상세 팝업에서 상세 정보를 확인할 수 있습니다.

8.1 프롬프트 수정 및 테스트 비교 시나리오
운영 중인 AI 시황 서비스의 프롬프트를 수정하고, 기존 프롬프트와 생성 결과를 비교한 후 반영하는 절차에 대한 가이드입니다. 

1) 현행 프롬프트 확인
데이터브릭스 SQL Editor 또는 관리자 대시보드(스키마 브라우저, AI 시황데이터 관리)에서 현행 프롬프트를 조회합니다.

2) 개발환경에서 프롬프트 수정
프롬프트 수정 방법은 다양하게 존재합니다. 
- 데이터브릭스 SQL Editor에서 UPDATE문으로 PROMPT_TEXT를 수정
- 관리자 대시보드(AI 시황데이터 관리 &gt; 프롬프트)에서 해당 프롬프트 수정
- 데이터브릭스 /notebooks/data_ingest/05_seed_prompts.py의 해당 프롬프트 내용 변경 후 노트북 실행

위 세 방법의 경우 기존 프롬프트는 내용은 백업되지 않아 유실됩니다.

- 데이터브릭스 SQL Editor에서 INSERT문으로 수정할 프롬프트의 PROMPT_TEXT만 수정하여 새로 등록
모든 프롬프트는 _load_prompt 함수로 nh_ai_prd.cofig.PROMPT_CATALOG 테이블에서 로드합니다. 위 함수는 동일한 PROMPT_ID로 여러 행이 있어도, UPDATED_AT DESC 후 LIMIT 1을 실행하므로 항상 최신 1건만 가져옵니다. 따라서 실제로 노트북이 사용하는 것은 항상 UPDATED_AT이 가장 최신인 1건이므로 프롬프트 이력관리가 가능합니다.

8.2.3 Step 2: sns-int에 스케줄링 및 ADLS 전송 추가
sns-int는 내부 스케줄러 서버로, 정해진 주기로 sns-ext를 호출하고 수집 결과를 ADLS Gen2에 업로드합니다.
1) WebJob 모듈 생성
sns-int/webjobs/gemini/ 디렉토리에 실행 로직을 구현합니다.
예시:
# sns-int/webjobs/gemini/run.py
import os
import requests
import json
from datetime import datetime, timezone, timedelta
from utils.adls_helper import AdlsWriter

SNS_EXT_URL = os.getenv("SNS_EXT_URL")
SNS_API_KEY = os.getenv("SNS_API_KEY")

def main():
    """Gemini 수집기 실행 → ext 호출 → ADLS 업로드"""
    # 1. sns-ext 호출
    resp = requests.post(
        f"{SNS_EXT_URL}/gemini/theme-stocks",
        headers={
            "X-API-Key": SNS_API_KEY,
            "User-Agent": "sns-int/1.0.0"
        },
        json={"query": "오늘의 주요 테마 종목"},
        timeout=120
    )
    resp.raise_for_status()
    data = resp.json().get("data", [])

    # 2. ADLS에 JSONL 업로드
    now_kst = datetime.now(timezone(timedelta(hours=9)))
    writer = AdlsWriter(source="gemini")

    lines = [json.dumps(item, ensure_ascii=False) for item in data]
    content = "\n".join(lines)

    writer.upload(
        content=content,
        program="theme_stocks",
        ingest_date=now_kst.strftime("%Y%m%d"),
        hour=now_kst.strftime("%H"),
        suffix=".jsonl"
    )

    print(f"[Gemini] {len(data)}건 ADLS 업로드 완료")

if __name__ == "__main__":
    main()

8.4.2 단계별 대응절차
1) Bronze 테이블에서 이상 시점 전후 데이터 확인
Databricks SQL 또는 관리자 대시보드에서 이상이 발생한 시간대의 KRX1 데이터를 조회하여, 언제부터 데이터가 깨졌는지 확인합니다. 중점적으로 확인할 사항은 다음과 같습니다. 
- 정상 데이터와 이상 데이터의 비교 쿼리(예: 특정 종목(삼성전자 005930)의 시간대별 데이터 확인)

- raw_data 원본 길이 확인 쿼리
기존 KRX1 고정폭 합계: 89자 (CODE 4 + SHRN_ISCD 9 + BSOP_DATE 8 + BSOP_HOUR 6 + STCK_PRPR 12 + PRDY_VRSS_SIGN 1 + PRDY_VRSS 12 + PRDY_CTRT 6 + ACML_VOL 15 + ACML_TR_PBMN 15 + HOUR_CLS_CODE 1)

2) 수집 서버에서 원본 JSONL 데이터 확인
udp_collector 서버에 접속하여, ADLS에 업로드되기 전 로컬 버퍼 또는 ADLS landing의 원본 JSONL을 확인합니다.

# 최신 JSONL 파일 내용 확인

  "event_ts": "2026-03-23T09:30:15.123456",
  "port": 30101,

  "packet_id": "UDP_30101_1711158615123456_10200130",

3) 스키마 정합성 확인
스키마 정합성 확인 후 파싱 스키마 수정이 필요할 경우, 두 곳의 코드를 수정합니다.

- Databricks Bronze 적재 노트북
notebooks/data_ingest/00_ingest_bronze.py의 specs_30101을 수정합니다. 
예시:
# 수정 후
specs_30101 = [
    ('CODE', 4, 'string'),
    ('SHRN_ISCD', 9, 'string'),
    ('BSOP_DATE', 8, 'string'),
    ('BSOP_HOUR', 6, 'string'),
    ('STCK_PRPR', 15, 'string'),        # ← 15자로 변경
    ('PRDY_VRSS_SIGN', 1, 'string'),
    ('PRDY_VRSS', 12, 'string'),
    ('PRDY_CTRT', 6, 'string'),
    ('ACML_VOL', 15, 'string'),
    ('ACML_TR_PBMN', 15, 'string'),
    ('HOUR_CLS_CODE', 1, 'string'),
]

새로운 필드가 추가된 경우:
specs_30101 = [
    # ... (기존 필드 동일) ...
    ('HOUR_CLS_CODE', 1, 'string'),
    ('NEW_FIELD', 4, 'string'),         # ← 새 필드 추가
]

- 수집 서버 파서 수정
udp_collector/udp_listener/parsers/stock_parser.py의 _parse_krx1_packet() 함수도 동일하게 수정합니다. 이 파서는 수집 서버의 진단/모니터링 용도로 사용되며, Bronze 적재에는 직접적으로 관여하지 않지만 일관성을 위해 같이 수정하는 것이 좋습니다. 이 경우 수집서버의 파일을 직접 교체, 수정 후 서비스를 재시작 해야합니다.
stock_parser.py 수정:
# stock_parser.py의 _parse_krx1_packet 내 슬라이스 오프셋 수정
# 예: data[27:39] → data[27:42] (STCK_PRPR 12자 → 15자)

수집 서버 (udp_collector) 재시작:
sudo systemctl restart udp-listener

# 서비스 상태 확인
sudo systemctl status udp-listener

# 실시간 로그 확인
journalctl -u udp-listener -f

수집 서버의 udp_collector는 systemd 서비스(udp-listener.service)로 운영되며, 재시작 시 run.sh를 통해 실행됩니다. 재시작 후 60초 주기로 포트별 수신 통계가 로그에 출력되므로, 포트 30101의 수신 건수가 정상적으로 증가하는지 확인합니다.
동일한 패턴으로 다른 UDP 포트의 스키마 변경에도 대응할 수 있습니다.
